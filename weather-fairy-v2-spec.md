# Weather-Fairy v2 — 설계 명세서

> **앱 이름:** 우산챙겨? (표시 이름) / `grab-umbrella` (영문 코드네임 — 리포·번들 ID·폴더명)
> 출퇴근길 날씨 알림 앱. v1(Vue PWA)을 RN 앱 + 경량 백엔드로 재구성한다.
> 이 문서는 Claude Code 핸드오프용 설계 명세다. 구현 전 "확정"과 "TODO(확정 필요)"를 구분해 읽을 것.

## 0. 핵심 컨셉

이 앱은 "하루 날씨"가 아니라 **사용자가 실제로 바깥에 있는 두 순간 — 출근 시각, 퇴근 시각 — 의 날씨**에만 집중한다. 출근하면 하루 종일 실내에 있으므로, 종일 예보보다 그 두 시점이 압도적으로 중요하다.

앱이 답하는 질문은 단 두 개:
- 오늘 출근할 때 우산 챙겨야 해?
- 옷은 어떻게 입어야 해? (어제 대비 체감 포함)

킬러 기능은 화면이 아니라 **푸시**다. 사용자가 앱을 안 켜도, 출발 30분 전에 "우산 챙기세요" 한 줄이 와야 한다.

## 1. 기술 스택 (확정)

| 영역 | 선택 | 비고 |
|---|---|---|
| 앱 | React Native (Expo) | v1은 Vue PWA였음. 네이티브 푸시 확보가 전환 이유 |

| 푸시 | Expo Push | |
| 디자인 | Apple HIG 준수 | §7 |

**로그인 없음.** 계정 기반이 아니라 기기 기반. 식별자는 Expo 푸시 토큰.

## 2. 데이터 모델 (확정)

로그인이 없으므로 테이블 하나로 충분하다. 기기당 한 줄.

```
devices
─────────────────────────────────────────
push_token        TEXT  PK   -- Expo 푸시 토큰. 익명 식별자 역할
home_nx           INT        -- 집 격자 X (기상청 LCC DFS)
home_ny           INT        -- 집 격자 Y
work_nx           INT        -- 회사 격자 X
work_ny           INT        -- 회사 격자 Y
commute_start     TEXT       -- 출근 시각 "0900" (HHmm, KST 기준)
commute_end       TEXT       -- 퇴근 시각 "1800"
last_synced_at    TIMESTAMP
created_at        TIMESTAMP
```

설계 노트:
- 위치의 "주인"은 **앱 로컬 저장소**(AsyncStorage). 사용자가 보고 수정하는 화면이 붙는 곳. DB는 푸시 발송을 위한 사본이다.
  - **이유:** 로그인이 없어 서버엔 "누구"라는 개념이 없고 "이 푸시 토큰을 가진 익명 기기"만 있다. 사용자 설정이 가장 자연스럽게 사는 곳은 그 사람 폰이다(오프라인에서도 보임, 앱 켜자마자 서버 왕복 없이 표시).
  - **DB에도 두는 이유:** 푸시는 사용자가 앱을 안 켜도 서버가 혼자 깨어나 위치·시각을 보고 발송해야 하므로, 서버도 사본을 반드시 가져야 한다. 선택이 아니라 푸시의 구조적 요구.
  - **동기화는 항상 로컬 → DB 단방향.** 사용자가 폰에서 바꾸면 `/sync`로 DB를 갱신. DB가 로컬을 덮어쓰지 않음 → "어느 쪽이 최신?" 충돌 고민이 없음.
  - **트레이드오프(의식적 수용):** 기기를 바꾸면 설정이 안 따라옴(새 기기=새 푸시 토큰). 로그인을 뺀 대가. "기기 교체 시 유지"가 필요해지면 그때 로그인 도입(§10 정신).
- 위경도→격자(nx,ny) 변환은 **서버에서** 수행(변환 로직을 한 곳에 모음). v1의 `useConvertXY`(LCC DFS) 로직을 Go로 포팅.
- 미세먼지용 측정소 좌표(TM)는 **지금 넣지 않는다.** 미세먼지는 부차적 기능이라 나중에 컬럼 추가.
- 중복 푸시 방지용 발송 로그가 필요(§5). 별도 테이블 또는 devices에 `last_morning_push_date` / `last_evening_push_date` 컬럼 추가로 처리.

## 3. API 엔드포인트 (앱 ↔ 서버) (확정)

총 3개. 로그인/회원 엔드포인트 없음.

### `POST /sync`
앱이 위치·출퇴근 시각·푸시 토큰을 서버에 올린다(설정이 바뀔 때마다 호출).
- 입력: push_token, 집/회사 위경도(lat/lng), commute_start, commute_end
- 서버: 위경도를 격자(nx,ny)로 변환 후 devices에 upsert
- 출력: 성공 여부

### `GET /forecast`
메인 화면용. "오늘/내일 출근·퇴근 시간대 날씨"를 **단기예보**로 조회해 가공된 결과 반환.
- 입력: 집/회사 격자좌표(또는 push_token으로 조회), 출퇴근 시각
- 출력: 출근 카드 / 퇴근 카드용 가공 데이터 (날씨 상태, 기온, 강수확률 등)
- **추가: 각 시각 주변의 시간별 흐름도 함께 반환** (§7.1 점진적 공개용). 출근 카드를 탭하면 펼쳐질 "출근 시각 전후 시간별 예보"를 같이 내려줌. 단기예보가 1시간 단위로 오므로 새 API 호출 없이 같은 응답에서 슬라이스만 하면 됨 → 추가 비용 거의 없음.
  - 출근: 출근 시각 이전을 더 보여줌(가는 길이 중요). 예: 출근 2시간 전 ~ 출근 후 1시간
  - 퇴근: 퇴근 시각 이후를 더 보여줌. 예: 퇴근 1시간 전 ~ 퇴근 후 2시간
  - (TODO) 정확한 윈도우 범위는 와이어프레임 시 확정

### `GET /forecast/now` (선택)
**초단기실황**으로 "지금 바깥 기온" 등. 없어도 동작하지만 메인 화면을 풍부하게 함.

## 5. Cron 푸시 흐름 (확정)

cron은 웹 서비스와 **별도 서비스**로 돌아 spin-down 영향을 받지 않는다.

```
1. cron이 매 N분(예: 10분)마다 깨어남  (트래픽 없어도 독립 실행)
        ↓
2. "지금부터 ~30분 뒤가 출근/퇴근"인 기기만 DB에서 SELECT
        ↓
3. 각 기기의 위치·시각으로 초단기예보 조회
   (출근=집 위치, 퇴근=회사 위치 / 6시간 내라 정밀)
        ↓
4. 한 줄 메시지로 압축 → Expo Push 발송
   ("비 와요, 우산 챙기세요" / "어제보다 추워요, 겉옷")
```

설계 노트:
- **왜 10분 간격:** 사용자마다 출근 시각이 제각각(7시/8시반/9시…). 자주 깨어나 "30분 뒤가 누군가의 출근인가" 확인하는 방식이 가장 단순. 10분 간격이면 실제 푸시는 출발 20-30분 전 사이 — 충분한 정밀도. 1분 간격 불필요.
- **중복 발송 방지:** 같은 출근건에 두 번 쏘지 않도록 "오늘 출근 푸시 보냄" 표시를 DB에 기록(§2의 last_*_push_date).
- **메시지 압축 규칙:** 예보 데이터 → "한 줄 + (선택)이모지". 우산 필요 여부 + 어제 대비 체감을 우선. (LLM 한 줄 생성은 선택적 고도화.) 알림 제목에 앱 이름 '우산챙겨?'를 살리면 자연스러움 (예: 제목 "우산챙겨?" / 본문 "오늘 퇴근길 비 와요").
- **"어제 대비 체감":** 절대 기온보다 "어제보다 추움/더움"이 옷 결정에 유용. 전일 동시간 기온과 비교.

## 7. 디자인 기준 (확정)

**Apple Human Interface Guidelines 준수.** 사용자가 아이폰 사용자라 iOS 기본 날씨 앱이 1차 레퍼런스.

원칙:
- 절제와 위계. 출근/퇴근 두 핵심 정보에 시각적 무게를 몰아줌(큰 카드). 미세먼지 등 나머지는 작게.
- 시스템 폰트(SF Pro), 표준 여백, 최소 색상.
- "한 화면 한 목적" — 출근 우산 여부가 한눈에.
- **점진적 공개(progressive disclosure):** 첫 화면은 두 순간(출근/퇴근)의 핵심만. 더 알고 싶으면 카드를 탭해 그 시각 주변의 시간별 흐름을 펼침. 종일 타임라인을 전면에 깔지 않음(그러면 일반 날씨 앱이 되어 정체성이 흐려짐).

### 7.1 화면 구성(초안)
- **메인:** 출근 카드 + 퇴근 카드 (두 핵심 시점). 각 카드에 우산 여부·기온·강수확률·어제 대비 체감을 압축. 하단에 보조 정보(선택: 지금 바깥 기온).
- **카드 탭 시(점진적 공개):** 해당 출퇴근 시각 전후의 시간별 예보를 펼침. 출근은 이전 쪽을 더, 퇴근은 이후 쪽을 더(비대칭). `/forecast`가 같은 응답에 이 데이터를 포함하므로 추가 호출 없음.
- **설정:** 집/회사 위치, 출퇴근 시각, 알림 on/off.

## 9. 구현 순서 (제안)

1. **DB + /sync** — devices 테이블, 위경도→격자 변환, upsert
2. **/forecast** — 단기예보 조회·가공, 캐싱, 메인 화면 데이터
3. **RN 앱 골격** — 설정 화면(위치/시각), 로컬 저장, /sync 연동
4. **메인 화면** — 출근/퇴근 카드 (Apple HIG)
5. **Expo Push 셋업** — 토큰 발급·등록
6. **Cron 푸시** — §5 흐름, 초단기예보, 중복방지
7. (고도화) 어제 대비 체감, 미세먼지, LLM 메시지

각 단계가 독립적으로 동작 확인 가능하도록 진행.
